# Content Research Writer

## Activation
When the user asks for help writing articles, blog posts, newsletters, case studies, or technical content with research and citations.

## Workflow

### 1. Understand the Project
Ask clarifying questions:
- Topic and main argument?
- Target audience?
- Desired length/format?
- Goal: educate, persuade, entertain, explain?
- Existing research or sources?
- Writing style: formal, conversational, technical?

### 2. Collaborative Outlining

```markdown
# Article Outline: [Title]

## Hook
- [Opening line/story/statistic]
- [Why reader should care]

## Introduction
- Context and background
- Problem statement
- What this article covers

## Main Sections

### Section 1: [Title]
- Key point A, B
- Example/evidence
- [Research needed: specific topic]

### Section 2: [Title]
- Key points
- Data/citation needed

## Conclusion
- Summary, call to action, final thought

## Research To-Do
- [ ] Find data on [topic]
- [ ] Source citation for [claim]
```

### 3. Research and Citations

Search for relevant information, find credible sources, extract key facts and data:

```markdown
## Research: [Topic]

Key Findings:
1. **[Finding]**: [Data] [1]
2. **[Finding]**: [Data] [2]
3. **Expert Quote**: "[quote]" - [Source] [3]

Citations:
[1] Author. (Year). "Title". Publication.
[2] Author. (Year). "Title". Publication.
```

### 4. Hook Improvement

Analyze and strengthen introductions:

**Option 1 (Data-driven):**
> "Last month, I asked AI to analyze 500 customer interviews. It took 30 minutes instead of 3 weeks."

**Option 2 (Question):**
> "What if you could talk to every customer before your morning coffee?"

**Option 3 (Story):**
> "Sarah spent two weeks building the wrong feature. Not because she didn't understand her users..."

Evaluate: Does it create curiosity? Promise value? Is it specific? Match the audience?

### 5. Section-by-Section Feedback

For each section, review:

```markdown
# Feedback: [Section Name]

## What Works Well
- [Strengths]

## Suggestions
### Clarity
- [Issue] -> [Fix]

### Flow
- [Transition issue] -> [Better connection]

### Evidence
- [Claim needing support] -> [Add citation]

### Style
- [Tone issue] -> [Match voice better]

## Specific Line Edits
Original: > [quote]
Suggested: > [improved]
Why: [explanation]
```

### 6. Voice Preservation
- Learn their style from existing writing
- Suggest, don't replace
- Match tone: formal, casual, technical
- Ask: "Does this sound like you?"

### 7. Citation Management

Support formats: inline, numbered references, footnotes.

Maintain running citations list:
```markdown
## References
1. Author. (Year). "Title". Publication.
2. Author. (Year). "Title". Publication.
```

### 8. Final Review

```markdown
# Full Draft Review

## Overall Assessment
Strengths, impact assessment

## Structure & Flow
Organization, transitions, pacing

## Content Quality
Argument strength, evidence, examples

## Pre-Publish Checklist
- [ ] All claims sourced
- [ ] Citations formatted
- [ ] Examples clear
- [ ] Transitions smooth
- [ ] Call to action present
- [ ] Proofread for typos
```

## Workflows
- **Blog Post**: Outline -> Research -> Draft sections with feedback -> Polish
- **Newsletter**: Hook ideas -> Quick outline -> Draft -> Review -> Polish
- **Technical Tutorial**: Outline steps -> Code examples -> Explanations -> Test -> Review
- **Thought Leadership**: Unique angle -> Research perspectives -> Strong POV -> Evidence -> Conclusion
